一、基座模型
Qwen2.5-7B-Instruct

二、数据集
核心原则： 工业级客服助手必须以真实业务对话为主，再辅以高质量公开数据提升泛化性。
1）自有业务数据
从历史客服记录中清洗高质量闭环对话，要求：助手回复准确、专业、无违规内容。

每条对话补充明确的 system prompt，例如：
“你是 XX 公司的智能客服助手，需耐心、准确回答用户关于产品退换货、物流查询等问题。如果无法解决，请礼貌引导用户转人工。”

数据量：精标 1,000 ~ 5,000 条多轮对话，即可看到明显的风格和知识注入效果。

2）公开客服/对话数据集
JDDC 2020
CCPC（中文客服对话）
BELLE 指令数据

3）混合策略建议
将业务数据与公开数据按 3:1 ~ 5:1 的比例混合，让模型既学到领域知识，又不会丢失通用对话能力。
所有数据统一转换为 SWIFT 要求的格式


三、模型评估指标
1）自动化基础指标（快速迭代用）
ROUGE-L：衡量生成回复与标准答案的字面重叠度，适合初筛。

BERTScore：基于预训练模型计算语义相似度，比 ROUGE 更关心含义一致性，尤其适合中英文。

Perplexity（PPL）：监控基座语言模型能力是否退化，仅作参考。

2）强 LLM 裁判评估（重要）
同一批验证集 prompt，抽 200~500 条分别输入原始基座、LoRA 微调后模型，调用强模型（GPT4o / 通义千问 4）做裁判打分二选一，自动统计：
Win Rate（微调优于原模型比例）：客服场景目标≥75%
Tie 持平 / Lose 劣于原模型
评判维度固定 5 项（裁判打分标准）：
业务答案准确性
客服语气专业友好度
多轮上下文理解能力
负面情绪安抚效果
无幻觉、合规不越权


维度	说明	评分（1-5）
解决率	回复是否直接解决了用户的问题	5：完全解决
准确性	提供的产品信息、政策是否准确无误	5：完全准确
满意度	语气是否友好、同理心、主动安抚	5：非常满意
安全性	无隐私泄露、无攻击性、无幻觉承诺	扣分项
转人工时机	遇到能力外问题能否及时引导转人工	5：正确引导

3）上线前（上线前最终判定，半量化打分 1-5 分）
这些指标只能通过人工评测或 A/B 测试获得：

首响解决率（FCR）：用户首次提问就被彻底解决的比例（客服场景最重要指标）。

转人工率：低于阈值说明助手接管了大部分工作，但也要平衡满意度。

用户满意度评分：事后收集用户打分（1~5 星），目标是 ≥ 4.5。

平均对话轮次：轮次减少表示效率提升。

风险拦截率：输出是否合规（如承诺赔偿、辱骂等），需配安全评测集。

随机抽取 200 条验证对话人工评测，计算平均分≥4 分才算合格：
首轮问题解决率（核心业务指标）：首次回答即可解决用户问题，目标≥80%
转人工触发合理性：无法回答时主动引导人工，不强行乱答
用户情绪处理：投诉、不满回复安抚自然
多轮上下文记忆：不遗忘前文用户提供订单、诉求信息
简洁度：无冗余废话，客服回复简短高效

四、参考实现链路
1）项目名称
基于 Qwen2.5-7B+ms-swift 多领域通用客服 LoRA 微调系统
2）实现
面试可梳理的完整工程链路
公开数据集清洗 → 统一对话格式与人设 prompt → 4bit QLoRA 显存优化训练 → 基于 PPL 防过拟合监控 → 本地轻量化自动化评估 → LoRA 推理与权重合并交付。

前置信息
数据：ECD 电商客服数据集；基座：Qwen2.5-7B-Instruct；框架：ms-swift；显卡 3090 24G；目标：通用客服 LoRA
步骤 1：梳理 ECD 原始文件结构
ECD 解压后一般分三个文件：train.txt dev.txt test.txt
每行格式（制表符 \t 分割）：
分类标签 \t 用户多轮拼接文本 \t 客服回复
示例单行：
0 快递什么时候到？可以加急吗？ 亲亲发货后2-3天送达，暂不支持加急哦
步骤 2：写转换脚本，转 ms-swift 标准 jsonl
新建 ecd2swift.py，直接复制运行，作用：
统一通用客服 system prompt（适配多行业通用客服定位）
丢弃无用分类标签，拼接用户所有问句
生成一行一条 jsonl 训练格式
同时分出训练集、验证集（9:1 划分）
执行完成后目录出现 train.jsonl、val.jsonl。
步骤 3：抽样控制数据总量（MVP + 简历项目关键，不要全量 ECD）
ECD 完整几十万条，全量训练耗时极长，只取 2000 条以内足够做演示项目，训练速度快、不易过拟合。
修改脚本里读取逻辑，在 append 后加限制：
重新运行脚本，总样本锁定 2000 条，训练 1-2 小时即可跑完。
步骤 4：ms-swift 一键 QLoRA 训练命令（3090 24G 专用）
参数说明（面试可讲优化点，简历写亮点）：
4bit QLoRA：显存占用 10G 左右，单卡稳定不 OOM；
gradient_checkpointing：大幅降低激活显存开销；
rank=16、epoch=2：小数据集防止过拟合；
每 100 步在验证集计算 PPL，监控收敛。
骤 5：训练过程观察关键指标（求职项目核心评估，无需付费 API）
训练日志持续输出 val_ppl：
合格标准：微调后验证集 PPL 明显低于原始基座；
过拟合判断：训练集 PPL 持续下降，验证集 PPL 后期上升 → 选用更早保存的 checkpoint。
步骤 6：训练完成后本地交互式测试
挑选最优 checkpoint（val_ppl 最低的文件夹），交互式对话看客服效果：
输入各类测试问题：
物流：快递什么时候发货，能加急吗
售后：衣服掉色可以全额退款吗
边界问题：能给我额外补偿优惠券吗
记录对比【原基座回答】vs【LoRA 微调后回答】，截图放进简历项目演示。
步骤 7：轻量化自动化评估（简历加分，自主实现无第三方付费）
写简易评估脚本完成 3 项量化指标：
验证集批量 BERTScore（本地 bert-base-chinese，衡量回答语义匹配度）
违规关键词检测（统计私自承诺赔付、补偿的比例）
人工抽样 50 条样本统计问答合格率
步骤 8：导出合并完整模型（项目交付闭环）
